## Анализ предложений

### ✅ Реализовать — высокий приоритет

#### 1. Исправление схемы подписок (FCM)

**Почему обязательно:**  
Текущая схема `user_subscriptions` имеет `PRIMARY KEY (fcm_token)` — это значит, что один пользователь может подписаться **только на одну** группу/преподавателя. При повторной подписке на другую сущность старая **затирается** (`ON CONFLICT DO UPDATE`). Это критический баг, а не фича.

**Что делаем:**
- Изменить PK на композитный: `PRIMARY KEY (fcm_token, entity_type, entity_id)`
- Добавить инвалидацию мёртвых FCM-токенов после `SendEachForMulticast` (проверять `IsRegistrationTokenNotRegistered`)
- Добавить лимит подписок на токен (макс. 10) для защиты от абьюза

---

#### 2. Пагинация по неделям (серверная)

**Почему нужно:**  
Сейчас фронт получает **всё расписание** (весь год/семестр — массив `[]Day`) и фильтрует в `ScheduleContent.tsx` локально. Это работает, но:
- Избыточный трафик: клиент скачивает ~50KB+ на каждый запрос, хотя показывает 1 неделю
- Нет кэширования по неделям — React Query кэширует весь блоб, и при навигации по неделям не может кэшировать каждую неделю отдельно
- Невозможно поделиться ссылкой на конкретную неделю

**Что делаем:**
- `?week_start=YYYY-MM-DD` query-параметр. Если не передан — поведение как сейчас (совместимость)
- Фильтрация на бэкенде из L1/L2 кэша (данные уже в памяти, это просто `for` цикл по `[]Day`)
- В ответ добавить метаданные: `week_start`, `week_end`, `has_prev`, `has_next`
- Фронт: React Query кэширует каждую неделю отдельно (`queryKey: ['schedule', type, id, weekStart]`)
- Фронт: prefetch следующей недели в фоне
- URL отражает неделю: `/schedule/group/123?week=2025-09-01`

---

#### 3. Экспорт в iCal (.ics)

**Почему нужно:**  
Это одна из самых запрашиваемых функций в любом расписании. Позволяет:
- Импортировать расписание в Google Calendar / Apple Calendar / Outlook
- Подписаться на живое обновление (webcal://)
- Видеть расписание университета рядом с личными событиями

**Технические нюансы:**
- Используем библиотеку `arran4/golang-ical` (лёгкая, zero-dependency)
- Формируем VEVENT из каждого занятия с полями: SUMMARY (дисциплина), LOCATION (аудитория), DESCRIPTION (преподаватель, подгруппа)
- Кэширование сгенерированного .ics в памяти с TTL 30 мин — не пересчитывать на каждый запрос
- Правильные заголовки: `Content-Type: text/calendar`, `Content-Disposition`
- Защита: query-параметр `?token=...` (простой статичный токен из конфига, не авторизация)

### ✅ Реализовать — средний приоритет

#### 4. Расширенные уведомления (дайджест + напоминания)

**Анализ:**  
Тип A (изменение расписания) уже работает. Предлагаемые типы B, C, D имеют смысл, но с оговорками:

| Тип | Полезность | Сложность | Вердикт |
|-----|-----------|-----------|---------|
| **B — Расписание на завтра** | 🟢 Высокая | Средняя | ✅ Реализуем |
| **C — Напоминание перед парой** | 🟡 Средняя | Средняя | ✅ Реализуем, но опционально |
| **D — Утренний дайджест** | 🔴 Низкая | Средняя | ❌ Дублирует тип B |

**Почему убираем тип D:**
Утренний дайджест дублирует вечерний (тип B) — пользователь уже знает расписание на сегодня с вечера. Два одинаковых пуша утром и вечером раздражают и ведут к отписке. Лучше один качественный дайджест вечером.

**Что делаем:**
- Расширить таблицу `user_subscriptions` полями настроек
- Scheduler (горутина с тикером раз в минуту) внутри syncer
- Логика дайджеста: кэш → фильтр по завтрашней дате → формирование текста → FCM
- Логика напоминания: тикер → проверка ближайших пар → отправка за N минут

## Технические решения

### Пагинация: почему серверная, а не клиентская?

Текущая архитектура уже фильтрует `[]Day` на фронте (строка 341 в `ScheduleContent.tsx`). Перенос на сервер даёт:
1. **Меньше трафика** — клиент получает ~7 дней вместо ~180
2. **Лучший кэш** — React Query кэширует каждую неделю отдельно, prefetch делает навигацию мгновенной
3. **Shareable URLs** — ссылка на конкретную неделю
4. **Нет overhead на сервере** — данные уже в L1 кэше, это просто `for` цикл

### iCal: почему кэшируем?

Генерация .ics — это сериализация ~180 занятий в текстовый формат. Операция лёгкая (~1ms), но при массовом запросе (Calendar apps polling каждые 15 мин) может создать ненужную нагрузку. TTL 30 мин — разумный компромисс.

### Scheduler: почему внутри syncer, а не отдельный сервис?

Проект — это монолит на одном бинарнике. Выносить scheduler в отдельный процесс — overengineering. Горутина с `time.Ticker` внутри syncer идеально вписывается в архитектуру.
