# Стратегия и план тестирования omsu-mirror

Этот документ описывает подходы, инструменты и конкретные сценарии тестирования для обеспечения стабильности, производительности и корректности работы системы.

---

## 1. Бэкенд (Go)

### 1.1 Модульное тестирование (Unit Tests)
**Инструменты:** Стандартный пакет `testing`, `testify` (опционально).

- **Парсинг Upstream API (`internal/upstream`):**
    - Тест: Десериализация JSON из `SAMPLE_RESPONSE.md` в модели `Lesson` и `Day`.
    - Ожидание: Корректная обработка всех полей, отсутствие ошибок при наличии `null` в необязательных полях.
- **Поисковый индекс Trie (`internal/cache`):**
    - Тест: Добавление данных с разным регистром и спецсимволами (ё/е).
    - Тест: Поиск по префиксу (например, "мбс" -> "МБС-501").
    - Ожидание: Мгновенный возврат результатов, игнорирование регистра, корректная работа нормализации.
- **Логика кэширования и хранения (`internal/cache`, `internal/storage`, `internal/config`):**
    - Тест: Сохранение и получение данных из `sync.Map` (`memory_test.go`).
    - Тест: Работа с GZIP-кэшем, инвалидация записей и сбор статистики.
    - Тест: CRUD операции в SQLite репозиториях (справочники, расписания, инциденты) с использованием `in-memory` базы данных (`dict_repo_test.go`, `schedule_repo_test.go`, `incident_repo_test.go`).
    - Тест: Парсинг переменных окружения и установка значений по умолчанию (`config_test.go`).

### 1.2 Интеграционное тестирование API
**Инструменты:** `fiber.App.Test()`.

- **Rate Limiting:**
    - Тест: Отправка 130 запросов в минуту с одного IP.
    - Ожидание: Первые 120 — статус 200, последующие — 429 с заголовком `Retry-After`.
- **Security Headers:**
    - Тест: Проверка наличия `X-Content-Type-Options`, `X-Frame-Options` в ответе.
- **ETag & 304 Not Modified:**
    - Тест: Повторный запрос с заголовком `If-None-Match`.
    - Ожидание: Статус 304 и пустое тело ответа.
- **Health & Incidents:**
    - Тест: Проверка структуры ответа `/health` и `/incidents`.

## 2. Фронтенд (React)

### 2.1 Тестирование состояния (Zustand)
**Инструменты:** `Vitest`, `React Testing Library`.

- **Favorites Store:**
    - Тест: Добавление/удаление из избранного, проверка сохранения в `localStorage`.
- **Settings Store (Theme):**
    - Тест: Переключение темы, проверка атрибута `data-theme` на `document.documentElement`.
- **Recent Searches:**
    - Тест: Добавление 6-го запроса.
    - Ожидание: В списке остается только 5 самых новых элементов.

### 2.2 Тестирование интерфейса (UI Components)
- **Schedule Rendering:**
    - Тест: Отображение карточки "Нет занятия" (placeholder) в пустых промежутках между парами.
    - Тест: Корректная цветовая индикация (Прак/Лаб) в зависимости от типа занятия.
- **Subgroup Filtering:**
    - Тест: Выбор подгруппы и проверка фильтрации списка уроков.

## 3. Сквозное тестирование (E2E)
**Инструменты:** `Playwright` или `Cypress`.

- **Критический путь пользователя:**
    1. Открытие главной страницы.
    2. Поиск группы "МБС-501".
    3. Переход на страницу расписания.
    4. Добавление в избранное.
    5. Возврат на главную и проверка наличия группы в быстром доступе.

---

## 4. Нагрузочное тестирование
**Инструменты:** `wrk` или `hey`.

- **Цель:** Проверка стабильности при 10,000+ RPS к закэшированным эндпоинтам.
- **Метрики:** RAM < 50MB, CPU < 10% под нагрузкой, P99 < 5ms.

## 5. Интеграция в CI
- Автоматический запуск `go test ./...` при каждом Pull Request.
- Проверка линтинга бэкенда (`golangci-lint`) и фронтенда (`eslint`).
