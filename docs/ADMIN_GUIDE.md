# Руководство администратора (Admin CLI)

Для управления проектом `omsu_setka` разработана специальная утилита командной строки — **Admin CLI**. Она позволяет быстро проверять состояние серверов, управлять синхронизацией и отслеживать инциденты.

## Быстрый старт

Утилита доступна как в корне проекта (скрипт `admin.sh`), так и внутри контейнера бэкенда (команда `admin`).

### Способ 1: Использование с хост-машины
Скрипт автоматически считывает `ADMIN_KEY` из файла `.env`.
```bash
./admin.sh status
```

### Способ 2: Использование внутри контейнера
Команда интегрирована в образ бэкенда.
```bash
docker exec -it omsu_mirror_backend admin status
```

---

## Доступные команды

| Команда | Описание |
| :--- | :--- |
| `status` | Проверка здоровья (health) и детального статуса синхронизации. |
| `sync` | Принудительный запуск фоновой синхронизации (справочники + расписания). |
| `incidents` | Просмотр последних проблем с доступностью оригинального сервера ОмГУ. |
| `logs` | Просмотр логов бэкенда в реальном времени (только снаружи). |
| `restart` | Быстрый перезапуск всех сервисов проекта (только снаружи). |

## Прямой доступ через API

Для интеграции с внешними системами мониторинга вы можете использовать HTTP-запросы напрямую. Все административные эндпоинты требуют заголовок `X-Admin-Key`.

### Пример ручной синхронизации:
```bash
curl -X POST http://localhost:8080/api/v1/sync/trigger \
-H "X-Admin-Key: your-secret-admin-key"
```

### Эндпоинты:
- `GET /api/v1/health` — Общее состояние.
- `GET /api/v1/sync/status` — Статистика синхронизации.
- `GET /api/v1/incidents` — История инцидентов.

## Автоматический деплой (CI/CD)

Проект настроен на автоматический деплой через **GitHub Actions**. При каждом пуше в ветку `main` или `master` происходит тестирование, сборка Docker-образов и обновление кода на сервере через SSH.

### Настройка GitHub Secrets

Чтобы деплой заработал, необходимо добавить следующие секреты в настройках вашего репозитория (`Settings > Secrets and variables > Actions`):

| Секрет | Описание |
| :--- | :--- |
| `SERVER_HOST` | IP-адрес или домен вашего сервера. |
| `SERVER_USER` | Имя пользователя для SSH (например, `root`). |
| `SSH_PRIVATE_KEY` | Приватный SSH-ключ для доступа к серверу. |
| `DEPLOY_PATH` | Путь к папке проекта на сервере (например, `/home/user/omsu_mirror`). |
| `SSH_PORT` | (Опционально) Порт SSH, если он отличается от 22. |

### Переменные окружения

Для безопасности конфиденциальные переменные приложения передаются через GitHub Secrets и динамически инжектируются в контейнеры при каждом деплое (они не сохраняются в файл `.env` на диске сервера):

| Секрет | Переменная окружения |
| :--- | :--- |
| `ADMIN_KEY` | Ключ администратора для доступа к API. |
| `CORS_ALLOWED_ORIGINS` | Разрешенные домены для CORS (например, `https://omsu.example.com`). |
| `VITE_FIREBASE_API_KEY` | API ключ из настроек Firebase. |
| `VITE_FIREBASE_PROJECT_ID` | Идентификатор проекта Firebase. |
| `VITE_FIREBASE_MESSAGING_SENDER_ID` | Идентификатор отправителя (Sender ID). |
| `VITE_FIREBASE_APP_ID` | Идентификатор приложения (App ID). |
| `VITE_FIREBASE_VAPID_KEY` | Публичный VAPID ключ для Web Push. |

## Особенности сборки (FCM)

Для работы Push-уведомлений ключи Firebase должны быть доступны **на этапе сборки** (build-time), так как они вшиваются в Service Worker. CI/CD пайплайн автоматически передает их через `build-args`. При ручной сборке Docker-образа фронтенда используйте аргументы `--build-arg`.

После добавления этих секретов любой пуш в основную ветку приведет к автоматическому обновлению приложения и его конфигурации на сервере без необходимости ручной настройки переменных окружения на хосте.
