# Руководство по разработке и поддержке ядра (Backend)

Этот документ содержит подробную информацию о бэкенде `omsu_setka`, его архитектуре и стратегиях обслуживания.

## Технологический стек

- **Язык:** Go 1.22+
- **Фреймворк:** Fiber v2 (самый быстрый HTTP-фреймворк на Go)
- **База данных:** SQLite 3 (режим WAL, через `modernc.org/sqlite` для чистой реализации на Go без CGO)
- **HTTP-клиент:** fasthttp (для работы с upstream без аллокаций)
- **Логирование:** zerolog (структурированные логи высокой производительности)
- **Документация:** Swagger (через `swaggo`)

## Архитектура системы

Бэкенд работает как высокопроизводительный BFF (Backend-for-Frontend) с трехуровневой стратегией кэширования:

1. **L1 (Память):** Использует `sync.Map` для хранения предварительно отрендеренных JSON-блоков. Позволяет отдавать данные мгновенно без повторной сериализации.
2. **L2 (Персистентность):** SQLite хранит сжатые (GZIP) данные для выживания при перезапуске.
3. **L3 (Upstream):** Оригинальный API ОмГУ, доступ к которому осуществляется через фоновую синхронизацию или «ленивую» загрузку (lazy-fetch).

### Фоновая синхронизация

- **Справочники:** Группы, преподаватели и аудитории синхронизируются периодически (от 12 до 24 часов).
- **Расписания:** «Горячие» расписания (запрашиваемые пользователями за последние 24 часа) синхронизируются каждые 15–30 минут.
- **Lazy-Fetch:** При отсутствии данных в кэше они загружаются по запросу, кэшируются и попадают в приоритет для фонового обновления.

### Поисковый индекс

В памяти используется **Trie (префиксное дерево)** для сверхбыстрого автодополнения (< 1 мс). Индекс перестраивается при каждом обновлении справочников.

## Процедуры обслуживания

### Мониторинг состояния
- **`/health`**: Возвращает детальную информацию о состоянии системы, включая время работы, статистику кэша (хиты/промахи) и доступность сервера ОмГУ.
- **`/incidents`**: Список последних событий доступности источника (история падений и восстановлений).

### Конфигурация
Все настройки управляются через переменные окружения. См. корень проекта `.env.example`. Ключевые параметры:
- `UPSTREAM_RATE_LIMIT`: Максимальное количество одновременных запросов к серверу ОмГУ.
- `RATE_LIMIT_GENERAL`: Ограничение запросов к API mirror (по умолчанию 120/мин).
- `CACHE_SCHEDULE_TTL`: Время жизни кэша расписаний.

### Работа с данными
- **`CleanExpired()`**: Периодически удаляет устаревшие записи из базы данных SQLite.
- **Прогрев L1**: При запуске «горячие» записи загружаются из SQLite в память для устранения задержек при первом обращении.

## Разработка

### Установка
```bash
cd core
go mod download
```

### Запуск локально
```bash
go run cmd/server/main.go
```

### Обновление Swagger
Если изменены хэндлеры API, обновите документацию:
```bash
swag init -g cmd/server/main.go
```

### Обработка ошибок
- Ошибки upstream автоматически логируются в таблицу `upstream_incidents` и доступны через API для фронтенда.
- Используйте структурированные логи для отладки высоконагруженных участков кода.
